브랜치: test/Tank/WebRTC
작성일: 2026-03-24

1. 개요

현재 브랜치는 탱크 제어 프로세스를 하나의 통합 실행 흐름으로 구성한다.
VR 틸팅 경로는 다음 요소들로 이루어진다.

- 휴대폰 브라우저 IMU 센서 취득
- HTTP 제어 및 WebRTC 영상 시그널링
- PTZ 모드 전환 및 IMU 필터링
- PCA9685 기반 서보 각도 출력
- 컨트롤러 버튼으로 VR 시작/정지, 0점 조절을 요청하는 UI 명령 브리지

메인 실행 파일:
- src/mqtt_tank_drive.cpp

주요 관련 모듈:
- src/http_vr_server.cpp
- src/ptz_control.cpp
- src/vr_remote_input.cpp
- src/web_rtc_stream.cpp
- web/tilt_vr/app.js


2. 전체 실행 구조

2.1 메인 프로세스 구성

메인 프로세스는 아래 구성요소를 초기화한다.

- 탱크 모터 제어
- PTZ 컨트롤러
- 카메라 캡처
- RTSP 서버
- WebRTC 스트림 서버
- HTTP VR 서버
- 컨트롤러 입력 스레드
- MQTT 주행 제어 루프
- RC 상태 퍼블리셔

각 모듈의 역할은 다음과 같다.

- CameraCapture: libcamerasrc로부터 BGR 프레임을 수집
- WebRtcStreamServer: BGR 프레임을 GStreamer webrtcbin으로 전달하여 브라우저에 송출
- HttpVrServer: 웹 파일 제공, PTZ 모드 API, IMU API, WebRTC 시그널링 API, 컨트롤러 UI 명령 API 제공
- PtzController: IMU 값을 필터링하고 pan/tilt 서보 목표 각도로 변환
- VrRemoteInput: Linux evdev 입력을 읽고 일부 버튼을 UI 제어 명령으로 변환


2.2 데이터 흐름

현재 VR 틸팅 제어 흐름은 다음과 같다.

휴대폰 IMU
-> web/tilt_vr/app.js
-> POST /imu
-> PtzController::handle_imu()
-> 필터링된 pan/tilt 목표값 계산
-> 서보 스무딩 스레드
-> PCA9685 PWM 출력
-> SG90 팬/틸트 구동

현재 영상 흐름은 다음과 같다.

libcamerasrc
-> CameraCapture
-> WebRtcStreamServer::push_bgr_frame()
-> GStreamer appsrc
-> videoconvert
-> v4l2h264enc
-> rtph264pay
-> webrtcbin
-> 브라우저 RTCPeerConnection
-> 숨겨진 video 엘리먼트
-> 좌/우 캔버스로 동일 영상 복제 렌더링

컨트롤러 UI 명령 흐름은 다음과 같다.

Linux 입력 이벤트
-> VrRemoteInput
-> VrUiCommandBridge 시퀀스 증가
-> GET /vr/ui-commands
-> 브라우저 polling
-> connect()/disconnect() 또는 zeroCalibrate()


3. PTZ 모드 구조

PTZ는 두 개의 모드를 가진다.

- manual
- vr

manual 모드:
- MQTT PTZ 명령을 수신
- pan_left / pan_right / tilt_up / tilt_down

vr 모드:
- 휴대폰 IMU 값을 수신
- IMU 입력이 일정 시간 이상 끊기면 자동으로 manual 모드로 복귀

타임아웃 조건은 개념적으로 다음과 같다.

if dt_since_last_imu_ms > imu_timeout_ms
then mode = manual

이 구조는 오래된 IMU 값이 계속 서보를 움직이지 않도록 하기 위한 안전장치이다.


4. IMU 처리 수학 구조

4.1 원시 입력값

브라우저는 아래 센서 값을 읽는다.

- pitch = beta
- roll = gamma
- yaw = alpha

이를 시간 t에서 다음과 같이 표기할 수 있다.

raw_pitch(t), raw_roll(t), raw_yaw(t)


4.2 브라우저 측 0점 조절

0점 조절을 실행하면 현재 IMU 값을 baseline으로 저장한다.

baseline_pitch = raw_pitch(t0)
baseline_roll  = raw_roll(t0)
baseline_yaw   = raw_yaw(t0)

이후 서버로 보내는 재중심화 값은 다음과 같다.

pitch_sent = wrap180(raw_pitch - baseline_pitch)
roll_sent  = wrap180(raw_roll  - baseline_roll)
yaw_sent   = wrap180(raw_yaw   - baseline_yaw)

여기서

wrap180(x):
  while x > 180, x = x - 360
  while x < -180, x = x + 360

즉, 웹의 0점 조절 버튼과 컨트롤러 버튼으로 실행하는 0점 조절은 모두 브라우저 내부 baseline을 다시 잡는 동작이다.


4.3 급격한 값 변화 제거

필터링 전에 비정상적인 1샘플 점프를 제거한다.

예를 들어:

abs(wrap180(pitch - prev_pitch)) > reject_pitch_threshold 이면 샘플 폐기
abs(wrap180(roll  - prev_roll )) > reject_roll_threshold  이면 샘플 폐기
abs(wrap180(yaw   - prev_yaw  )) > reject_yaw_threshold   이면 샘플 폐기

이 단계는 센서 튐 현상을 줄이기 위한 것이다.


4.4 저역통과 필터

Pitch와 Roll은 지수평활 기반 저역통과 필터를 적용한다.

filtered_pitch(k) = (1 - a) * filtered_pitch(k-1) + a * guarded_roll(k)
filtered_roll(k)  = (1 - a) * filtered_roll(k-1)  + a * guarded_pitch(k)

여기서:

- a = kImuAlpha = 0.2

주의:
현재 휴대폰 장착 방향 기준으로 구현이 의도적으로 축을 교차 사용한다.

- 휴대폰 roll -> tilt 경로
- 휴대폰 pitch -> pan 경로


4.5 Yaw unwrap 및 필터링

Yaw는 각도 주기성이 있으므로 +-180도 경계를 직접 비교하면 불안정해진다.
그래서 먼저 unwrap을 수행한다.

raw_delta = wrap180(yaw(k) - yaw(k-1))
guarded_delta = clamp(raw_delta, -yaw_max_step, +yaw_max_step)
yaw_unwrapped(k) = yaw_unwrapped(k-1) + guarded_delta

그 다음 1차 저역통과 필터를 적용한다.

yaw_alpha = dt / (tau + dt)
filtered_yaw(k) = filtered_yaw(k-1) + yaw_alpha * (yaw_unwrapped(k) - filtered_yaw(k-1))

여기서:

- tau = kYawFilterTauSec = 0.2 s
- dt는 IMU 샘플 간 시간차이며 최소/최대 범위로 clamp됨


5. Pan/Tilt 매핑 수식

5.1 일반 축 매핑

코드에서는 IMU 각도를 서보 각도로 선형보간하여 변환한다.

clamped = clamp(value, -imu_max_deg, +imu_max_deg)

if clamped >= 0:
  servo = lerp(center, positive_limit, clamped / imu_max_deg)
else:
  servo = lerp(center, negative_limit, (-clamped) / imu_max_deg)

여기서

lerp(a, b, t) = a + (b - a) * t


5.2 Pan 계산

Pan은 pitch 기반 pan 값과 yaw 기반 pan 값을 혼합한다.

pitch_pan = map(filtered_roll * pan_sensitivity_scale)
yaw_pan   = map(filtered_yaw * yaw_pan_gain * pan_sensitivity_scale)

blended_pan = 0.35 * pitch_pan + 0.65 * yaw_pan

final_pan = clamp(blended_pan, pan_center - vr_pan_limit, pan_center + vr_pan_limit)

사용 상수:

- pan_sensitivity_scale = 0.8
- yaw_pan_gain = 1.35
- vr_pan_limit = 60 deg

해석:
- pan 방향에서는 yaw 기여도가 더 큼
- 중심 기준 최대 회전 범위를 제한하여 과도한 회전을 방지


5.3 Tilt 계산

Tilt는 filtered pitch 계열 값으로부터 계산된다.

tilt_target = map(filtered_pitch * tilt_sensitivity_scale)
final_tilt = clamp(tilt_target, tilt_center - vr_tilt_limit, tilt_center + vr_tilt_limit)

사용 상수:

- tilt_sensitivity_scale = 0.8
- vr_tilt_limit = 50 deg


5.4 서보 스무딩

서보는 목표 각도로 즉시 점프하지 않고, tick당 최대 이동량 제한을 둔다.

next_angle = step_toward(current_angle, target_angle, max_step_per_tick)

여기서:

step_toward(c, t, s):
  if c < t: min(c + s, t)
  else:     max(c - s, t)

사용 상수:

- 업데이트 주기 = 50 Hz
- tick당 최대 각도 변화 = 3 deg

따라서 최대 각속도는 대략:

3 deg/tick * 50 tick/s = 150 deg/s

이 구조는 서보 떨림과 급격한 움직임을 줄이기 위한 것이다.


6. WebRTC 영상 구조

이 브랜치에서는 VR 웹 영상 기본 경로를 MJPEG가 아니라 WebRTC로 전환했다.
브라우저는 아래 순서로 세션을 맺는다.

browser createOffer()
-> POST /webrtc/session
-> 서버가 SDP offer 파싱
-> GStreamer webrtcbin이 answer 생성
-> 서버가 SDP answer 반환
-> 브라우저가 setRemoteDescription(answer) 수행

GStreamer 파이프라인 개념은 다음과 같다.

appsrc
-> videoconvert
-> I420
-> v4l2h264enc
-> h264parse
-> rtph264pay
-> webrtcbin

이 방식은 MJPEG보다 대역폭 효율이 높고, 모바일 브라우저 기반 실시간 영상에 더 적합하다.


7. 이번 브랜치에서 추가한 컨트롤러 버튼 매핑

7.1 추가된 브리지 구조

새로운 공유 브리지를 추가했다.

- include/vr_ui_command_bridge.hpp
- src/vr_ui_command_bridge.cpp

이 브리지는 두 개의 단조 증가 시퀀스 카운터를 가진다.

- session_toggle_seq
- zero_calibrate_seq

브라우저는 아래 API로 이를 polling한다.

- GET /vr/ui-commands


7.2 버튼 매핑

현재 버튼 매핑은 다음과 같다.

- BTN_SIDE  -> VR 세션 시작/정지 토글
- BTN_LEFT  -> 0점 조절
- KEY_VOLUMEUP    -> 속도 증가
- KEY_VOLUMEDOWN  -> 속도 감소

동작 순서는 다음과 같다.

1. BTN_SIDE 입력
   -> VrRemoteInput이 toggle 요청 발생
   -> VrUiCommandBridge가 session_toggle_seq 증가
   -> 브라우저가 새로운 시퀀스 값 감지
   -> 현재 연결 안 되어 있으면 connect()
   -> 현재 연결 중이면 disconnect()

2. BTN_LEFT 입력
   -> VrRemoteInput이 zero 요청 발생
   -> VrUiCommandBridge가 zero_calibrate_seq 증가
   -> 브라우저가 새로운 시퀀스 값 감지
   -> VR 연결 상태이면 zeroCalibrate() 실행


7.3 중요한 제약 사항

컨트롤러 버튼은 휴대폰 센서에 직접 접근하지 않는다.
버튼은 브라우저에 "웹 버튼과 동일한 동작을 해라"라고 요청하는 구조이다.

따라서:

- 0점 조절은 폰 웹 페이지가 열려 있고 polling 중일 때만 동작
- VR 시작/정지도 폰 웹 페이지가 열려 있을 때만 동작
- 일부 모바일 브라우저, 특히 iPhone 계열은 최초 1회 모션 센서 권한 허용에 터치가 필요할 수 있음

이는 서버 수학 로직의 한계가 아니라 브라우저 보안 제약 때문이다.


8. 실제 운용 순서

권장 운용 순서는 다음과 같다.

1. 휴대폰 브라우저에서 /web/ 접속
2. 최초 1회 센서 권한이 뜨면 직접 허용
3. 컨트롤러 BTN_SIDE를 눌러 VR 세션 시작
4. 휴대폰을 원하는 기준 자세로 둠
5. 컨트롤러 BTN_LEFT를 눌러 0점 조절
6. 이후 휴대폰 움직임으로 팬/틸트 제어 수행

추적 기준이 틀어졌다면:

- BTN_LEFT를 다시 눌러 재영점 수행

세션을 끄고 싶다면:

- BTN_SIDE를 다시 눌러 종료


9. 현재 설계의 의미

현재 VR 틸팅 구조는 다음 역할 분리로 설계되어 있다.

- 브라우저: 센서 권한 처리와 baseline 관리
- 서버: 필터링, 좌표 변환, 목표 각도 계산
- 서보 스레드: 부드러운 실제 각도 추종
- 컨트롤러: 브라우저 UI 동작을 원격으로 요청

이 구조의 장점은 다음과 같다.

- 브라우저가 센서 권한 문제를 직접 처리
- 서버는 일관된 수학 연산과 구동만 담당
- 컨트롤러는 웹 UI를 대체하는 보조 입력 장치로 작동

11. 향후 개선 제안

1. 0점 조절 baseline을 브라우저가 아니라 서버로 이동
   - 휴대폰은 raw IMU만 전송
   - 컨트롤러 0점 조절이 완전히 서버 기준으로 동작

2. /vr/ui-commands polling 대신 WebSocket 또는 data channel 사용
   - 더 낮은 지연
   - 반복 HTTP 요청 감소

3. toggle 기반이 아니라 명시적 버튼 분리
   - VR 시작
   - VR 종료
   - 0점 조절

4. WebRTC에 STUN/TURN 지원 추가
   - 더 넓은 네트워크 환경에서 안정성 향상
